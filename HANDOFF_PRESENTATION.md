## 1. Что такое docAgent (в одном абзаце)

docAgent — это веб-инструмент для **анализа и сравнения деловых документов**
(`.docx`, `.pdf`, `.md`) — договоров, регламентов, политик — с помощью
**self-hosted LLM (Qwen3)**. Пользователь загружает документ(ы), формулирует
фокус анализа обычным текстом и получает связный аналитический отчёт в Markdown
с возможностью копирования и экспорта. Ключевая особенность: модель запускается
**в собственном периметре** (Ollama / vLLM / LM Studio), поэтому конфиденциальные
документы **не уходят во внешние облачные сервисы**.

## 2. Проблема и ценность

**Боль:** сравнение двух версий многостраничного договора или регламента вручную
— это медленно, утомительно и чревато пропусками. Юрист/комплаенс тратит часы,
сверяя пункты «было/стало», и всё равно может упустить значимое изменение срока,
суммы или условия ответственности. Использовать публичные облачные LLM
(ChatGPT и т.п.) для этого **нельзя** — документы конфиденциальны.

**Ценность docAgent:**
- Связный отчёт о различиях за минуты, с акцентом на **смысловые** изменения
  (права/обязанности, сроки, суммы, штрафы, ответственность, расторжение).
- Произвольный **фокус анализа** обычным языком («обрати внимание на сроки оплаты
  и штрафы»).
- **Данные не покидают периметр** — критично для юр./фин./гос. документов.
- Помимо сравнения — **анализ одного документа** (вопрос-ответ по содержанию или
  структурированный обзор) и **конвертация в Markdown**.

## 3. Сценарии использования (что умеет UI)

Веб-интерфейс (vanilla JS, одна страница) с двумя вкладками:

1. **⇄ Сравнение документов** — загрузить версию A и версию B, задать фокус,
   получить отчёт о различиях. Можно скачать каждый документ как Markdown.
2. **🔍 Анализ документа** — загрузить один документ, задать вопрос (или оставить
   пустым для общего структурированного обзора), получить ответ.

Дополнительно в обеих вкладках:
- **Конвертация в Markdown** любого загруженного файла (с подсчётом токенов).
- **Экспорт отчёта** (.md) и **копирование** в буфер.
- Индикатор здоровья модели сверху (доступна / нет, какая модель, какой endpoint).
- Прогресс-индикатор с человекочитаемыми статусами по ходу анализа.

## 4. Архитектура

```
[1–2 файла .docx/.pdf/.md]
        │
   extractor (markitdown) → плоский Markdown → секции по заголовкам + таблицы
        │
   chunker (оценка токенов через tiktoken, +5% запас на расхождение с Qwen)
        │
   ┌────┴───────────────────────────┐
влезает в контекст          не влезает в контекст
   │                                 │
comparator: 1 запрос        aligner: сопоставление секций (fuzzy по заголовкам)
   │                                 │
   │                        comparator: N попарных запросов → сведение в summary
   └────────────┬────────────────────┘
                │
        llm_client (Qwen3 через OpenAI-совместимый API; срезает <think>)
                │
          отчёт (Markdown)
                │
   FastAPI: /api/compare, /api/analyze, /api/convert, /api/health
                │
        web/index.html (рендер Markdown + санитизация DOMPurify)
```

**Модули (Python, ~10 файлов в `core/` + `server.py`):**

| Модуль | Назначение |
|---|---|
| `core/extractor.py` | `.docx/.pdf/.md` → Markdown (markitdown) → секции по заголовкам |
| `core/chunker.py` | Оценка токенов (tiktoken), проверка «влезает целиком», группировка |
| `core/aligner.py` | Сопоставление секций двух версий (fuzzy-match заголовков, rapidfuzz) |
| `core/comparator.py` | Оркестрация: выбор режима, сборка отчёта, бюджеты токенов |
| `core/llm_client.py` | OpenAI-совместимый клиент к Qwen3; mock-режим; срез `<think>` |
| `core/prompts.py` | Системные/пользовательские шаблоны промптов (на русском) |
| `core/config.py` | Конфиг из `.env` (endpoint, модель, окно контекста, лимиты) |
| `server.py` | FastAPI: статика + REST-эндпоинты |
| `web/index.html` | Фронт на vanilla JS, две вкладки, экспорт |

## 5. Ключевое техническое решение: два режима сравнения

Главный инженерный вызов PoC — **ограниченное контекстное окно** локальной модели.
docAgent выбирает режим автоматически:

- **whole** — если оба документа + инструкции + резерв под ответ влезают в окно
  контекста, сравнение делается **одним запросом** → максимальная связность и
  точность (модель «видит» оба документа целиком).
- **sectioned** — если не влезают: документы режутся на секции по заголовкам,
  версии A и B **сопоставляются** (fuzzy-match по заголовкам, rapidfuzz), затем
  сравниваются **попарно**, и посекционные заметки **сводятся в итоговый отчёт**
  отдельным запросом (map-reduce по сути).

Дополнительно для устойчивости:
- **+5% запас** к оценке токенов (tiktoken недооценивает токены Qwen на 3–5%).
- Динамический бюджет: `max_context − резерв под ответ − overhead промпта`,
  длинные секции усекаются под бюджет.
- **DoS-лимит** `MAX_SECTION_PAIRS` (по умолчанию 300) — битый/вредоносный docx
  с тысячами абзацев не породит тысячи запросов к LLM.

## 6. Приватность и безопасность

Это сильный аргумент для экспертной группы — стоит вынести на отдельный слайд.

- **Self-hosted модель.** LLM (Qwen3) работает в собственном периметре через
  Ollama / vLLM / LM Studio. Документы **не отправляются** во внешние облака.
- **Санитизация вывода.** Ответ модели рендерится как Markdown и чистится через
  **DOMPurify** — `<script>` / `<img onerror>` из ответа или из самого документа
  не выполнятся. Имена файлов экранируются перед вставкой в DOM.
- **Валидация входа.** Лимит размера файла (`MAX_FILE_MB`, проверка размера ДО
  чтения в память), допустимые расширения, лимит числа секций.
- **Граница (честно):** эндпоинты **без аутентификации** — PoC рассчитан на
  локальный запуск (`127.0.0.1`). Для сети нужен обратный прокси с авторизацией;
  `/api/health` раскрывает имя модели и base_url.

## 7. Технологический стек

- **Backend:** Python, FastAPI, openai SDK (OpenAI-совместимый клиент к Qwen3).
- **Извлечение текста:** markitdown (docx/pdf/md → Markdown), rapidfuzz
  (сопоставление секций), tiktoken (оценка токенов).
- **Модель:** Qwen3 (например `qwen3:8b`) через Ollama / vLLM / LM Studio.
- **Frontend:** одна HTML-страница, vanilla JS, DOMPurify + Markdown-рендер.
- **Тесты:** pytest — юнит-тесты (извлечение, токены, выравнивание, без сети) и
  e2e через FastAPI TestClient с **замоканным** LLM (запущенная модель не нужна).
- **Запуск:** `python server.py` (или `run.bat` на Windows — сам создаёт venv,
  ставит зависимости, поднимает сервер, открывает браузер).
- **mock-режим** (`MOCK_LLM=true`) — UI можно демонстрировать без живой модели.

## 8. Статус: что это PoC (границы — обязательно проговорить)

**Готово и работает:**
- Полный путь: загрузка → извлечение → выбор режима → отчёт → экспорт.
- Оба режима сравнения (whole / sectioned), анализ одного документа, конвертация.
- Поддержка `.docx`, `.pdf`, `.md`.
- Покрытие тестами ядра и e2e-путей.
- Защита от переполнения контекста, DoS-лимиты, санитизация.

**Чего нет (это PoC, не продакшн):**
- Нет аутентификации/авторизации, многопользовательского режима, аудита.
- Нет хранения истории сравнений / версионирования.
- Сопоставление секций — эвристика (fuzzy по заголовкам); сложные перестановки
  или документы без заголовков выравниваются хуже.
- Качество отчёта зависит от выбранной локальной модели и размера её контекста.
- Нет точной привязки к номерам страниц/пунктов исходника (работа идёт с текстом).
- Не извлекаются картинки/диаграммы; сложная вёрстка PDF может теряться.

## 9. Возможный roadmap (для слайда «что дальше»)

- Аутентификация + развёртывание в сети организации (за прокси).
- Более качественное выравнивание (семантическое, не только по заголовкам).
- Подсветка различий на уровне предложений/«дельта-вью», а не только текстовый отчёт.
- История и аудит сравнений; экспорт в .docx/.pdf.
- Оценка качества на реальном корпусе договоров; подбор оптимальной модели/окна.
- Пакетная обработка, очередь задач, асинхронность для больших документов.
